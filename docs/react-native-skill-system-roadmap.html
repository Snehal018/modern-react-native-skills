React Native skill system roadmap

A proposed pathway for expanding the current three-skill system into a practical agent operating model for modern React Native development. The goal is more predictable agent work, not a large catalog of overlapping instructions.

Current system Expansion model Skill roadmap Priority path Review gates
Recommendation

Add skills in layers: first a router and lifecycle skills, then domain-specific build skills, then release and observability skills. Keep shared standards in references so each skill stays short and deterministic.

Baseline

What exists now

The repo currently has a solid foundation: setup, feature delivery, review, and one shared standard. The next work should fill high-frequency gaps without duplicating the shared React Native rules.

Turns an existing Expo/RN boilerplate into the standard architecture.

foundation project setup

Adds feature modules while respecting existing app conventions.

feature work implementation

rn-quality-review

Reviews RN code against architecture, state, component, data, and type standards.

review quality

Model

How to expand without sprawl

Use skills for repeatable processes with distinct invocation triggers. Use shared references for standards that many skills need. Avoid creating a new skill for every library unless it changes the agent's process.

Split by process, not package

A package alone is not a skill. Create a skill when the agent needs a distinct sequence: navigation design, auth flow, offline sync, app release, performance diagnosis, or test strategy.

Keep a single source of truth

Shared standards should live in skills/_shared/. Individual skills should point to the relevant branch and add only process-specific steps.

Prefer model-invoked for core work

Core RN skills should be model-invoked because future agents need to discover them automatically. Niche or experimental skills can be user-invoked or routed.

Add a router when the catalog grows

Once there are more than seven to nine RN skills, create a router skill that chooses the right workflow and prevents agents from stacking unrelated skills.

Roadmap

Proposed future skills

The table is ordered by value and likely reuse. The first wave should make everyday app work safer; later waves cover release, native integration, and specialized workflows.

Skill Use when Why it earns a skill Priority
rn-skill-router The user asks broadly for React Native help and the agent must pick a workflow. Reduces cognitive load once the system has many skills and avoids wrong-skill starts. High
rn-navigation-flow Adding or changing routes, tabs, stacks, deep links, auth gates, or route params. Navigation failures are common and cross-cut routing, params, state, and UX. High
rn-api-data Designing API clients, query hooks, mutations, optimistic updates, or cache behavior. TanStack Query, Zod, services, loading states, and RN focus/online behavior need a repeatable path. High
rn-forms-validation Building forms, schema validation, field errors, submit flows, or edit/create screens. Forms combine UI, validation, mutation state, accessibility, keyboard behavior, and types. High
rn-auth-secure-storage Implementing login, token refresh, session persistence, protected routes, or secret storage. Auth has security pitfalls and must coordinate SecureStore, API clients, navigation, and cache clearing. High
rn-testing-strategy Adding tests for hooks, components, API boundaries, navigation flows, or regression coverage. Future agents need a reliable test selection loop, not ad hoc snapshots or shallow tests. High
rn-performance-pass Diagnosing slow screens, list jank, unnecessary renders, image cost, or animation issues. Performance work needs measurement, hypotheses, list rules, memoization restraint, and verification. Medium
rn-ui-system Creating or extending reusable UI primitives, design tokens, theming, or component variants. Prevents scattered styling and over-generic components while keeping app-specific UI coherent. Medium
rn-offline-sync Adding offline reads, queued mutations, persistence, conflict handling, or retry flows. Offline behavior is a distinct architecture branch with storage, query persistence, and product tradeoffs. Medium
rn-observability Adding error reporting, performance monitoring, logs, breadcrumbs, or release health checks. Instrumentation should be deliberate and privacy-aware, not scattered through components. Medium
rn-release-readiness Preparing EAS builds, app versions, store submissions, environment config, or release checks. Release work has a predictable checklist and high cost for missed steps. Medium
rn-native-module-bridge Adding native modules, config plugins, permissions, native views, or platform-specific code. Native work needs stronger guardrails than ordinary feature work and should inspect platform files first. Later

Sequence

Recommended build path

Build in waves. Each wave should leave the system useful on its own and should not require future skills to make the current ones coherent.

Wave 0

Polish the foundation

Complete

Add examples of ideal skill invocation, decide whether to add agents/openai.yaml, and create a short contribution gate for future skills. Do not add more behavior yet.

Wave 1

Cover everyday app work

Create rn-navigation-flow, rn-api-data, rn-forms-validation, and rn-auth-secure-storage. These are the highest-return skills because they appear in most product features.

Wave 2

Add quality loops

Create rn-testing-strategy and rn-performance-pass. These should integrate with rn-quality-review but keep their own diagnostic steps.

Wave 3

Add delivery operations

Create rn-release-readiness and rn-observability after the app-building workflows are stable. These skills are valuable but should not distract from feature correctness.

Wave 4

Add specialized branches

Add native, offline, maps, media, notifications, and animation skills only after real project demand proves the branch is frequent enough to justify its context load.

Governance

Review gates for every new skill

Each future skill should pass these gates before it lands. The intent is to keep the system predictable as it grows.

A new skill should change agent behavior. If it only restates the shared standard or lists package docs, make it a shared reference instead.

  • Has a distinct leading word and model-invoked trigger.
  • Defines when to use and when not to use it.
  • Starts with inspection and explicit preconditions.
  • Uses ordered steps with checkable completion criteria.
  • Points to shared standards instead of duplicating them.
  • Includes validation commands or concrete skip criteria.
  • States what agents must avoid.
  • Has no placeholder text, broad motivation, or sample clutter.
